feat(mcp): surface OAuth lifecycle as toasts (Phase 22a)

`mcp.oauth_required` and `mcp.oauth_completed` were sitting in
`IGNORED_EVENTS` — when an MCP server needed sign-in the user got
no feedback and the connection silently stayed dead. Wire them into
`sessionsStore.applyToRecord` next to the existing model-change
toast (side-effectful, so not part of the pure reducer):

- `mcp.oauth_required` -> info toast naming the server with a hint
  to use the Library panel. Doesn't auto-open the OAuth URL — the
  SDK already drives elicitation; auto-launching browsers on
  background events would be hostile UX.
- `mcp.oauth_completed` -> success toast.
- De-duplication via `SessionRecord._toastedOauthRequests: Set<string>`
  keyed on `<sessionId>:oauth:<requestId>` so resumed sessions don't
  replay stale notifications and stray `_completed` events from
  other clients (e.g. CLI driving the OAuth flow) are ignored.

4 new tests in `sessionsStore.mcpOauth.test.ts`. Exposed
`handleEvent` from the store return as `applySessionEvent` so tests
can exercise the full event pipeline without spinning up
`onSessionEvent` (same function the runtime subscription uses).

Fixture maintenance for the new `_toastedOauthRequests` field:
Playground.vue, sessionCommands.test.ts, boundedEvents.test.ts.

Gates: lint clean, 476 bun tests pass (+4 new), 68/70 smoke (the
pre-existing 08-audit-rehydrate flake also fails on plain main).
