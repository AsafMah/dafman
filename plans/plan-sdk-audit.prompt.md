# Plan — SDK + CLI feature audit (2026-05-22)

> Authoritative cross-reference of every surface exposed by
> `copilot-sdk-supercharged` and the bundled `@github/copilot` CLI
> against what Dafman currently implements. Generated by reading the
> SDK's `README.md` end-to-end (lines 1–1135), `dist/generated/rpc.d.ts`
> (~2400 lines), and the CLI's `changelog.json` (212 versions,
> 436 "added" entries).
>
> Use this doc when planning the next Phase. STATUS.md tracks "what
> shipped"; this doc tracks "what the platform offers and we haven't
> wired yet".

---

## Methodology

1. **Supercharged README** — every config option, every `client.*` /
   `session.*` method, every event, every handler.
2. **`dist/generated/rpc.d.ts`** — every RPC method on both the
   server-scoped (`createServerRpc`) and session-scoped
   (`createSessionRpc`) surface.
3. **`@github/copilot` CHANGELOG** (`changelog.json`) — features
   shipped by the CLI itself that surface to clients through one of
   the SDK channels above. Skimmed every "added" entry from v0.0.329
   through v1.0.48.
4. **Cross-reference** against `src-bun/app/` and `src/` for what's
   actually wired today.

## D. Truly new surfaces not previously in our plans

Concepts we hadn't considered before.

1. **`rpc.workspaces.*` (session-scoped workspace files API)** — a per-
   session file area with `listFiles` / `readFile` / `createFile`. The
   CLI uses this for plans (`plan.md`), checkpoints, paste-overflow
   spill (CLI 0.0.397: >30 KB clipboard pastes spill to workspace
   files). A "Files" tab on the chat could surface this directly.
2. **`rpc.commands.handlePendingCommand`** — the CLI can defer a slash
   command back to the *client* to handle. Lets the agent emit a
   command we render natively (e.g. "open the diff for these files").
3. **MCP server *instructions*** (CLI 0.0.400) — MCP servers can carry
   instructions that get injected into the system message. We should
   render these in the MCP registry detail panel.
4. **`subagentStart` hook** (CLI 1.0.7) — inject context into a sub-
   agent's prompt. Used by Critic / Rubber-duck.
5. **`preCompact` hook** (CLI 1.0.5) — run a command before context
   compaction. Could let us inject a "save plan to disk" step.
6. **`agentStop` / `subagentStop` hooks** (CLI 0.0.401, 1.0.45).
7. **`PermissionRequest` hook** (CLI 1.0.16) — programmatic
   approve/deny based on rules. Could power a "remembered permission"
   layer that runs *before* our PendingRequestCard.
8. **`postToolUseFailure` hook** (CLI 1.0.15) — only fires on tool
   failure. Could power an automatic "retry with different args" loop.
9. **Tool search (Claude only)** (CLI 1.0.6) — model discovers and uses
   tools dynamically per turn. Implications for our `availableTools`
   UI (it's hint, not hard cap).
10. **HTTP hooks** (CLI 1.0.35) — hook POSTs JSON to a URL instead of
    running a local command. Could power webhook integrations
    (Phase 9 automations) without a scheduler.
11. **OTel "GenAI" semantic conventions** (CLI 1.0.45) — tool spans +
    `gen_ai.client.operation.duration` metric. Phase 11 telemetry
    surface is now well-defined.
12. **Plugin hook env vars** (`PLUGIN_ROOT`, `COPILOT_PLUGIN_ROOT`,
    `CLAUDE_PROJECT_DIR`, `CLAUDE_PLUGIN_DATA`) — when we eventually
    do plugin support, pre-canned env contract.

### Phase 24 — Session workspace files surface (~2 d)

**Scope clarification (investigated 2026-05-22, see DEVLOG entry):**
`rpc.workspaces.*` targets the session's **infinite-sessions state
directory** at `~/.copilot/session-state/{sessionId}/` — NOT the
user's repo cwd. Subdirs: `checkpoints/`, `files/` (agent-generated +
paste-overflow spills per CLI 0.0.397), `plan.md`. Access is
relative-path-only inside that dir; no enforcement of "files agent
can touch in the user's repo" — that's the **permission system's**
job (and our `PermissionRuleEditor` already owns it).

Phase 24 surfaces these dirs in the UI:

1. **Plan panel** inside the chat tab (collapsible right rail) — read
   + render `<workspacePath>/plan.md` via `rpc.plan.read`; markdown
   render; auto-refresh on `plan.changed` events. (0.5 d)
2. **Generated-files tab** — list `<workspacePath>/files/` via
   `rpc.workspaces.listFiles`; click to read content via
   `rpc.workspaces.readFile`; "Export to disk" button copies to
   user-chosen location. Useful for paste-overflow recovery + agent-
   authored artifacts. (1 d)
3. **Checkpoint browser** — list `checkpoints/` (auto-snapshots
   from CLI 0.0.389+); pair with `rpc.history.truncate` for
   point-in-time rewind. (0.5 d)

Out of scope for Phase 24: anything claiming to "show what files the
agent is allowed to touch in your repo." That doesn't exist as an SDK
surface — it's deny-by-default permission prompts, full stop.

## G. Open questions

1. **Tool search (Claude only).** If a model dynamically discovers tools
   per turn, our `availableTools` UI is a *hint*, not a hard guarantee.
   Document this prominently in the tool toggle popover.
2. **Background tasks vs streaming.** `rpc.tasks` runs an agent in the
   background — does the streaming event channel still deliver
   `task_complete` for those? Verify before wiring the tasks panel.
3. **`session.foreground` / `background` semantics.** Server-mode
   only or available to embedded clients? Affects how we wire the
   Sessions Manager.
4. **`requestElicitation` capability negotiation.** Multi-client: if
   another client (e.g. a VS Code ACP host) joins our session, does
   our elicitation card win? Need to test with multi-client.
5. **`responseFormat: "image"`** — confirm via empirical test that the
   string is `"image"` not e.g. `"png"`/`"image_base64"`.
