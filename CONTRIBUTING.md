# Contributing to Dafman
Thanks for considering a contribution! Dafman is still early — there is a lot of room to shape the architecture and product. This guide tells you how to file issues, propose changes, and ship a PR.
## Code of Conduct
Participation in this project is governed by the [Code of Conduct](CODE_OF_CONDUCT.md). By participating, you agree to uphold it.
## Where to start
- Read [`plans/plan-overview.prompt.md`](plans/plan-overview.prompt.md) for the vision and principles.
- Read [`plans/plan-roadmap.prompt.md`](plans/plan-roadmap.prompt.md) to see what milestone we''re in.
- Browse open issues, especially those labeled `good first issue` and `help wanted`.
## Filing issues
- Use the appropriate issue template (bug or feature).
- Be specific about reproduction steps, the OS, the Rust + Node versions, and what you expected vs. what happened.
- For security issues, follow [`SECURITY.md`](SECURITY.md) — do not open a public issue.
## Development setup
```bash
git clone https://github.com/<owner>/dafman.git
cd dafman
npm install
npm run tauri dev
```
Run the test suites:
```bash
# frontend
npm run build      # vue-tsc type-check + vite build
npm test           # vitest unit + component tests
# backend
cd src-tauri
cargo test
cargo clippy -- -D warnings
cargo fmt --check
```
## Branching & commits
- Branch from `main`.
- Use short, descriptive branch names: `feat/multi-account-auth`, `fix/dark-mode-bubbles`, `docs/roadmap-update`.
- Conventional commits encouraged but not required. Examples:
  - `feat: per-session GitHub auth pinning`
  - `fix(chat): handle session.idle before final message`
  - `docs(plans): integrate Supercharged v2.0 features`
- Keep commits focused. Squash before merge unless history is meaningful.
## Pull requests
- Fill out the PR template.
- Reference any related issue.
- Include screenshots / GIFs for UI changes.
- Make sure CI is green.
- Update relevant files in `plans/` if you''re changing direction.
- Add or update tests for new behavior — see [`plans/plan-testingStrategy.prompt.md`](plans/plan-testingStrategy.prompt.md).
## Code style
- **Rust:** `cargo fmt`, `cargo clippy -- -D warnings`. No `unwrap` in command paths.
- **TypeScript:** strict mode. `vue-tsc --noEmit` must pass.
- **Vue:** components are dumb; data + actions live in composables / Pinia stores.
- **CSS:** prefer PrimeVue tokens (`var(--p-*)`); avoid hardcoded hex except for per-session accents.
## Architecture conventions
- Backend domains expose traits + impls (see [`plans/plan-architecture.prompt.md`](plans/plan-architecture.prompt.md)). Only `ipc/` and `app/` know about `tauri::*`.
- Frontend stores call IPC; components don''t.
- Every Tauri command returns `Result<T, AppError>`.
- Every event payload has a typed mirror on the frontend (eventually generated by `tauri-specta`).
